## Context

The dispatcher system prompt is built by `buildRulesSegment()` in `spec-teams-utils.ts`. It currently produces five rules, none of which mention tool-checking. The `## Agents` catalog (built by `buildAgentCatalogSegment()`) at the bottom of the prompt contains each agent's `**Tools:**` line, but there is no cross-reference from routing instructions to this catalog.

An audit of the full dispatcher prompt found 11 routing instructions across the Lifecycle, Explore Relay Protocol, and General Tasks sections — all saying "dispatch the most suitable available agent" without verifying tool availability. The verify agent is the most critical case: it's intentionally read-only (no `write`/`edit`), and the only hint is "read-only inspection" in the verify phase block, which doesn't explicitly connect to the catalog.

## Goals / Non-Goals

**Goals:**
- Add a single rule to the Rules section that instructs the dispatcher to check the `## Agents` catalog for tool availability before dispatching
- Make the rule concrete with examples that help the LLM connect task types to tool requirements
- Keep the addition minimal — one rule, no per-phase duplication

**Non-Goals:**
- Per-phase tool requirement lists (the catalog already has this information)
- Restructuring prompt section order
- Adding model/thinking information to routing decisions
- Changing agent `.md` files or agent parsing logic

## Decisions

### Decision 1: Single rule in Rules section (Option E from exploration)

**Choice:** Add one bullet to `buildRulesSegment()` referencing the `## Agents` catalog.

**Rationale:** The Rules section is always present and naturally houses operational rules. A single rule with concrete examples is:
- Minimal prompt cost (2–3 lines vs. 25 lines for per-phase additions)
- Not fragile to agent tool-list changes (the catalog is regenerated each session)
- Provides enough specificity without duplicating the per-agent tool lists

**Alternatives considered:**
- Per-phase tool hints in lifecycle blocks — too verbose, duplicates catalog data, and would need updating if agent tools change
- Tool-checking rule without examples — too abstract for LLMs; concrete examples help connect task types to tool requirements significantly
- Restructuring prompt to put catalog before lifecycle — would break the progressive reading flow (Identity → Team → Lifecycle → Relay → Tasks → Rules → Catalog)

### Decision 2: Rule references catalog by section name

**Choice:** The rule says "check the ## Agents catalog" (using the section heading).

**Rationale:** The `## Agents` heading is stable and generated by `buildAgentCatalogSegment()`. Using the heading name creates a direct link the LLM can follow in the prompt, without hardcoding indices or positions.

## Risks / Trade-offs

- **Risk:** The rule is "soft" guidance — LLMs may skip it under complex routing → **Mitigation:** The concrete examples (`write`/`edit` for implementation, `read`/`grep` for investigation, `bash` for commands) make the rule specific and actionable rather than abstract
- **Risk:** Prompt length increase (~3 lines) → **Mitigation:** Negligible compared to the full prompt (~2000+ words); the efficiency gain from better routing outweighs the token cost
- **Risk:** Rule doesn't prevent all misroutings → **Mitigation:** Not intended to. It bridges the gap between the routing instructions and the catalog; it doesn't guarantee perfect routing, just makes tool availability a factor in the routing decision
